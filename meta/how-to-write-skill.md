# 如何写一个好的 Skill 文档

这是"写 skill 的 skill"，所有新增规范文档都应遵循本文定义的格式。

## 核心原则

1. 为 AI 而写，不是为人而写 —— 结构优先于叙述，结论优先于过程
2. 每条规范必须可执行 —— 能判断"符合"或"不符合"，不写模糊建议
3. 反例和正例同等重要 —— AI 需要知道什么不该做
4. 保持最小化 —— 一个文件只覆盖一个主题，宁可拆分不要堆砌
5. frontmatter 必填 —— 这是触发检索的唯一依据

## 文档模板

每个 skill 文件必须以 YAML frontmatter 开头：

```markdown
---
title: 规范名称
domain: coding/backend        # 所属领域路径
keywords: [关键词1, 关键词2]  # 用于触发检索的词，越具体越好
triggers: [场景描述1, 场景描述2]  # 什么情况下 AI 应该读这个文件
scope: 适用范围的一句话描述
---

# 规范名称

## 核心原则
（必填，3-5 条，每条一句话，AI 优先读取这里）

## 规范细则
（具体规则，可分小节）

## 反例（Anti-patterns）
（列出常见错误做法，说明为什么错）

## 代码示例
（可选，仅在文字描述不够直观时添加）
```

## frontmatter 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `title` | 是 | 文档标题，与 H1 保持一致 |
| `domain` | 是 | 所属领域路径，如 `coding/backend` |
| `keywords` | 是 | 触发词列表，用于 README 索引自动生成 |
| `triggers` | 是 | 场景描述，AI 遇到这些场景时应读此文件 |
| `scope` | 是 | 一句话说明适用范围 |

## 分类决策规则

问自己：我下次需要这个 skill 时，脑子里第一个词是什么？
- 那个词就是它的目录名
- 如果这个 skill 在多个领域都用得上 → 放 `meta/`
- 如果还不确定 → 先不建文件，在 README 用注释记下来，想清楚再写

## 文件命名规范

- 全小写，单词用连字符分隔：`api-design.md`
- 名称即主题，不加领域前缀：`auth.md` 而不是 `backend-auth.md`

## 更新规范

- 有新经验时，先判断能否追加到现有文件，能追加就不新建
- 新建文件后必须同步更新 `README.md` 的触发索引表
- frontmatter 的 keywords 变化时，同步更新 README 索引
