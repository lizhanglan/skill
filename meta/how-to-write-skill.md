# 如何写一个好的 Skill 文档

这是"写 skill 的 skill"，所有新增规范文档都应遵循本文定义的格式。

## 核心原则

1. 为 AI 而写，不是为人而写 —— 结构优先于叙述，结论优先于过程
2. 每条规范必须可执行 —— 能判断"符合"或"不符合"，不写模糊建议
3. 反例和正例同等重要 —— AI 需要知道什么不该做
4. 保持最小化 —— 一个文件只覆盖一个主题，宁可拆分不要堆砌

## 文档模板

```markdown
# 规范名称

## 核心原则
（必填，3-5 条，每条一句话，AI 优先读取这里）

## 规范细则
（具体规则，可分小节）

## 反例（Anti-patterns）
（列出常见错误做法，说明为什么错）

## 代码示例
（可选，仅在文字描述不够直观时添加）
```

## 文件命名规范

- 全小写，单词用连字符分隔：`api-design.md`
- 名称即主题，不加前缀：`auth.md` 而不是 `backend-auth.md`

## 目录规范

- 按领域分目录，不按技术栈分
- 目录名全小写：`coding/`、`meta/`
- 代码片段统一放在领域目录下的 `snippets/` 子目录

## 更新规范

- 有新经验时，先判断归属哪个现有文件，能追加就不新建
- 确实是新主题才新建文件，同时更新顶层 `README.md` 的索引
